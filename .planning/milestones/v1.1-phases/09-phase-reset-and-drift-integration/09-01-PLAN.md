**New member variables** (add after `displayRatioIndex` declaration, around line 99):
- `int clockBeatCount = 0;` -- counts clock edges within one LFO cycle
- `int prevRatioIdx = -1;` -- tracks ratio changes to reset beat counter
- `float crossfadeFrom = 0.f;` -- output value captured at moment of phase reset
- `float crossfadeProgress = 1.f;` -- 0.0 = just reset, 1.0 = crossfade complete
- `float crossfadeDuration = 0.003f;` -- 3ms default
- `float lastOutputVoltage = 0.f;` -- previous frame's output for crossfade capture
- `dsp::TExponentialFilter<float> freqSlew;` -- frequency smoother for mode transitions

**Constructor additions** (after existing `configOutput` and `updateDisplayBuffer` calls):
- `freqSlew.setLambda(20.f);` -- 50ms time constant (lambda = 1/tau = 1/0.05 = 20)
- `freqSlew.out = 0.7f;` -- matches default Rate param value

**Modify processClockInput()** -- RATE-04 and RATE-05:

1. In the cable-disconnect block (where `clockState = FREE`), add `clockBeatCount = 0;`

2. In the timeout block (where `clockState = FREE`), add `clockBeatCount = 0;`

3. Remove the unconditional `phase = 0.0;` at line 287 (currently resets on every edge regardless of ratio).

4. In the `clockEdgeCount == 1` (first edge) block: set `clockBeatCount = 0;`, capture crossfade (`crossfadeFrom = lastOutputVoltage; crossfadeProgress = 0.f;`), then `phase = 0.0;`. First edge always resets.

5. In the `else if (rawPeriod > 0.001f)` block (second edge onward), AFTER the existing EMA smoothing and state transitions, add division-aware phase reset logic:
   - Increment `clockBeatCount++`
   - Compute `currentRatioIdx` using `paramQuantities[RATE_PARAM]->getScaledValue()` and `std::round(knobNormalized * 14.f)`, clamped 0-14. Only compute if `(clockState == ACQUIRING || clockState == LOCKED) && smoothedPeriod > 0.f`.
   - If `currentRatioIdx != prevRatioIdx && prevRatioIdx >= 0`, reset `clockBeatCount = 1` (this edge counts as beat 1 of new ratio).
   - Update `prevRatioIdx = currentRatioIdx`.
   - Determine `shouldReset`: if `currentRatioIdx >= 0 && RATIO_TABLE[currentRatioIdx] < 1.f`, compute `int divisor = (int)std::round(1.f / RATIO_TABLE[currentRatioIdx])` and `shouldReset = (clockBeatCount >= divisor)`. Otherwise `shouldReset = true`.
   - If `shouldReset`: capture crossfade (`crossfadeFrom = lastOutputVoltage; crossfadeProgress = 0.f;`), then `phase = 0.0; clockBeatCount = 0;`

**CRITICAL ordering note:** The crossfade capture uses `lastOutputVoltage` (previous frame's output) rather than computing the current waveform value. This avoids needing morph/character values inside processClockInput() and produces a nearly identical result since phase changes by less than 0.001 between frames at LFO rates. See Research Pitfall 1.

**Modify process()** -- DISP-04 and DISP-05:

1. After computing `targetFreq` (the freq value from either free-running or clocked path), rename the existing `freq` to `targetFreq`. Apply frequency slew: `float freq = freqSlew.process(args.sampleTime, targetFreq); freq = std::fmax(freq, 0.001f);`

2. Compute `bool isClocked = (clockState == ACQUIRING || clockState == LOCKED) && smoothedPeriod > 0.f;` BEFORE the drift section (it is already implicitly computed by the if/else block -- extract it to a named variable).

3. In the drift section, replace `float driftScale = driftAmount * 0.075f;` with:
   ```
   float maxDrift = isClocked ? 0.02f : 0.075f;
   float driftScale = driftAmount * maxDrift;
   ```

4. After computing `outputVoltage = 5.f * sample` and BEFORE `outputs[OUTPUT].setVoltage(...)`, insert the crossfade blend:
   ```
   if (crossfadeProgress < 1.f) {
       crossfadeProgress += args.sampleTime / crossfadeDuration;
       if (crossfadeProgress >= 1.f) {
           crossfadeProgress = 1.f;
       } else {
           float mix = 0.5f - 0.5f * std::cos((float)M_PI * crossfadeProgress);
           outputVoltage = crossfadeFrom + mix * (outputVoltage - crossfadeFrom);
       }
   }
   ```

5. After the crossfade (and before or after setVoltage), store `lastOutputVoltage = outputVoltage;`

**Anti-patterns to AVOID (from research):**
- Do NOT crossfade on the phase variable -- only on output voltage
- Do NOT apply crossfade BEFORE computeMorphedWave -- it must happen AFTER
- Do NOT reduce drift to zero in clocked mode -- 2% preserves analog character
- Do NOT apply frequency slew to the phase reset itself -- only to the frequency value
