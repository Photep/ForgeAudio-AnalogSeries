<objective>
Fix BUG-01: the clock tracker permanently locks out when the incoming clock speeds up by
more than 3x. Every fast edge is rejected as an outlier AND each rejected edge already ran
`clockTimer.reset()` upstream, so the no-pulse timeout never fires — the module stays LOCKED
at the stale tempo forever.

Add a consecutive-outlier counter to `ClockTracker.hpp`: after a threshold of consecutive
outliers, treat it as a genuine tempo change and drop to ACQUIRING to re-learn. Pin the fix
with a RED→GREEN regression test that drives a sustained >3x SPEEDUP (the case the old code
fails) plus the fast-clock narrow slowdown band.

Purpose: SC1 — clock re-acquires after a >3x / <1/3x tempo jump, no permanent lockout.
