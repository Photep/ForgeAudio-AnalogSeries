## Summary

Phase 20.1 is a **mechanical repositioning + artwork-swap** on a single, well-isolated locus: the `AnalogLFOWidget` constructor in `src/AnalogLFO.cpp` (lines 1523–1586) plus one SVG strip operation on `res/AnalogLFO-fresh.svg`. There is **no DSP, no atomic-bridge, and no rendering-algorithm change**. Every locked anchor is in `20.1-UI-SPEC.md` / `ROADMAP.md`, and the existing code already uses the exact `mm2px(Vec(x,y))` + `createParamCentered/createInputCentered/createOutputCentered` idiom that accepts the new constants directly. The 14HP→18HP change is a pure value swap in ~22 call sites plus a display box geometry change.

The two genuinely non-trivial sub-tasks are: (1) **SVG bezel-strip** — the fresh panel double-draws static metal knob bodies that will collide with the widget-owned knobs; these are 5 short element runs (1–2 circles + indicator path each) per the line ranges below, and must be removed while *keeping* jack art and (optionally) the faint scallop tick / anchor rings; and (2) **WaveformDisplay refit** — the display rendering is already ~90% proportional to `box.size.x`/`box.size.y`, so it re-flows into the bigger 81.44×26.00mm box automatically; only a handful of **hardcoded px constants** (margins, font sizes, dot/bracket insets) need re-tuning so the larger glass reads as a deliberate refit rather than a stretched artifact.

**Primary recommendation:** Treat this as three sequential, independently-verifiable units — (A) SVG strip + promote/cleanup, (B) widget ctor anchor swap + RATE/PHASE type upgrade + hex-bolt verify, (C) display box resize + internal constant re-tune — then one in-Rack visual gate at 100%/200% zoom. Do NOT touch `WaveformDisplay::step()`, the atomic `display*` loads, or any DSP. Build with `make install RACK_DIR=../Rack-SDK`, which copies into the live `~/Library/Application Support/Rack2/plugins-mac-arm64/` install.

### Anti-Patterns to Avoid
- **Double-drawn knobs:** leaving the metal body circles in the SVG under widget-owned knobs (the entire point of D-01).
- **Re-deriving anchors:** parsing the SVG for positions — D-07 says use the hardcoded mm table. The SVG `cx/cy` values happen to match the table (e.g. line 79 cx=18 cy=87 = CHARACTER) and serve as a cross-check, not a parse source.
- **Touching DSP/atomic bridge:** `WaveformDisplay::step()`, `module->display*` atomics, RATIO_TABLE, swing math — all frozen.
- **Editing `plugin.json` width:** there is no width field (verified) — Rack derives it. Adding one is wrong (D-05).

## Common Pitfalls

### Pitfall 1: SVG strip removes too much (kills the screen/jacks) or too little (leaves ghost rings)
**What goes wrong:** Deleting whole element runs by eyeballing line numbers and accidentally removing the display glass, divider, or jack rings — or leaving the inner-ring/center circle so a faint metal disc shows under the live knob.
**Why:** The knob art is interleaved with scallop-tick paths and label paths; the "body" is only specific circles + the indicator path.
**How to avoid — exact strip targets (verified line-by-line in `res/AnalogLFO-fresh.svg`):**

| Knob | KEEP (optional, faint anchor/scallop) | **REMOVE (metal body + center + indicator)** |
|------|----------------------------------------|----------------------------------------------|
| MORPH (hero) | 45–60 (anchor rings r=17.5/20.5, scallop ticks, 3 dot markers), 61 (`r=11.60` soft shadow — keep or drop) | **62** (`r=11.00 url(#metalHero)` body), **63** (`r=6.82` inner ring), **64** (indicator path), **65** (`r=0.8` center dot) |
| CHARACTER | 67–77 (scallop ticks), 78 (`r=6.80` soft shadow) | **79** (`r=6.20 url(#metal)` body), **80** (`r=3.84` inner ring), **81** (indicator), **82** (center dot) |

| PHASE | 118–128 (ticks), 129 (shadow) | **130, 131, 132, 133** |
| Trimpots ×5 | 141–148, 154–161, 167–174, 180–187, 193–200 (scallop ticks) | body+indicator pairs: **140+149**, **153+162**, **166+175**, **179+188**, **192+201** (`r=2.9 url(#metal)` + `d=...d4a017` indicator) |

**MUST KEEP (do not touch):** line 36–44 (display chrome/screen/brackets), 39–40 (the static placeholder waveform polylines in the art — these sit *behind* the live `WaveformDisplay` which is opaque-backgrounded, harmless), 45–60 hero anchor scaffolding (keep — reads as the knob socket), all label paths (66, 83, 100, 117, 134, 139, 206, 209, 212, etc.), jack rings (150–151, 163–164, 176–177, 189–190, 202–203, 207–208, 210–211, 213–216 incl. OUTPUT ember rings), divider 204, bolts 15–26, rails 12–13, brand 27–34.
**Decision (Claude's discretion D-01):** Recommend **keeping** the soft-shadow circles (61/78/95/112/129) and scallop ticks — they read as a recessed socket under the live knob and match the Forge Noir look. **Removing** the `url(#metal)`/`url(#metalHero)` body, inner-ring, indicator, and center-dot is mandatory.
**Warning signs:** A visible hard metal ring that doesn't rotate with the knob = body not stripped. Missing display glass = over-stripped.

### Pitfall 2: Stale install / stale dist masks the change
**What goes wrong:** Build succeeds, Rack still shows 14HP. The verification screenshots are of the old panel.
**How to avoid:** `make clean && make && make install` (or `make dist`), confirm timestamp on the installed `res/AnalogLFO.svg`, fully restart Rack (Rack caches SVGs at load).
**Warning signs:** Panel width unchanged in Rack; deleted SVGs still present in install dir.

### Pitfall 3: Hex-bolt px math vs mm art mismatch
**What goes wrong:** Bolts use `RACK_GRID_WIDTH` pixels, art bolt holes are at mm 4.60/86.84. At 18HP they should coincide, but a 1HP rounding can offset.
**How to avoid:** After build, zoom 200% on each corner; the widget bolt must sit centered on the art bolt-hole circle. If offset, switch the bolt anchors to `mm2px(Vec(4.60,4.60))` etc. (matches D-09 corners) rather than the grid-pixel form.

### Pitfall 4: Display constants that are NOT proportional
**What goes wrong:** Bigger box, but text/dots/margins keep their absolute px size and look tiny / float in dead space — "stretched artifact" the user explicitly warned against.
**How to avoid:** See the re-tune list in §WaveformDisplay below. These are the only hardcoded values; everything else scales with `box.size`.

### Pitfall 5: Forgetting `box.size` is in PX, not mm, inside `WaveformDisplay`
**What goes wrong:** `box.size = mm2px(Vec(81.44,26.00))` means inside the widget `box.size.x` ≈ 308px, `box.size.y` ≈ 98px (mm2px ≈ ×3.7795). All internal `box.size.x * fraction` math is in px and re-flows correctly; but any *added* constant must be in px to match. Phase 20 box was mm2px(63.93,17.98) ≈ 241.6×67.96 px.
**How to avoid:** When re-tuning, reason in px (or in fractions of `box.size`), never raw mm.
