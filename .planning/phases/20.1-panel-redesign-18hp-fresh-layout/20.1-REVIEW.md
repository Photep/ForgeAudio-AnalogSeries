## Summary

This is a visual panel-redesign phase. The modified C++ surface is intentionally narrow: the `AnalogLFOWidget` constructor anchors (18HP mm coordinates), four `bg->box.pos` re-centering lines in the Forge knob/trimpot structs, `createWidgetCentered` hex-bolt placement, and a set of absolute-px tuning constants inside `WaveformDisplay`. DSP, `process()`, atomics, `RATIO_TABLE`, and enum bindings were not touched (PANEL-12 preservation), and spot checks confirm they are unchanged.

The high-risk item I traced carefully is the new `bg->box.pos = box.size.minus(bg->box.size).div(2)` centering line in all four control structs. It is **correct**: `setSvg()` (foreground) runs before the centering line and sizes the knob `box`; `bg->setSvg()` runs before it and wraps `bg->box.size` to the larger bg art; the bg SVG dimensions are confirmed larger than the foreground in every case (trimpot 6.30 > 5.40, hero 18.0 > 16.38, secondary 13.5 > 11.99), so the subtraction yields the negative offset that centers the larger art. Rack's `FramebufferWidget` sizes its framebuffer from the children bounding box, so the negative-offset bg is not clipped. The 7-iteration human in-Rack gate (Plan 05) independently confirmed correct rendering at 100%/200%.

Coordinate/binding integrity checks all pass: panel viewBox is `91.44 x 128.5` (18HP), corner bolts are vertically and horizontally symmetric (2.60 / 88.84 across 91.44; 2.60 / 125.90 across 128.5), left/right content margins match (7.70 / 83.74→7.70), and every trimpot column's attenuator parameter aligns with the CV-input column directly below it (MORPH / CHARACTER / DRIFT / FM / PHASE in both rows). All four SVGs are well-formed XML with no NaN/Infinity geometry.

No correctness, security, or data-loss defects were found in the modified code. The findings below are quality/maintainability items.

## Warnings

### WR-01: Dead, duplicated tuning constant `pillValueSize` is a silent-drift hazard

**File:** `src/AnalogLFO.cpp:1328`
**Issue:** `pillValueSize` is declared `[[maybe_unused]]` and never read. It documents itself as "BPM value" size, but `drawBpmStack()` carries its own independent `bpmFontSize = 4.0f` literal (line 1220). These are two separate sources of truth for the same visual constant. A future maintainer editing `pillValueSize` to retune the BPM pill will see no effect, because the live value lives in `drawBpmStack`. The `[[maybe_unused]]` attribute was added to silence the compiler warning rather than to fix the duplication, which entrenches the trap.
**Fix:** Either delete the dead declaration entirely, or make it the single source of truth and have `drawBpmStack` consume it:
```cpp
// In WaveformDisplay, promote to a member or pass through:
static constexpr float pillValueSize = 4.0f;  // BPM value pill size
...
// drawBpmStack:
float bpmFontSize = pillValueSize;  // single source of truth
```
Preferred: delete it, since `drawTextOverlays` does not draw the BPM value itself (it delegates to `drawBpmStack`). The comment on the line already admits "drawBpmStack owns its own bpmFontSize," which is the defect, not a justification.

## Info

### IN-01: Re-tuned px constants are scattered magic numbers across two functions

**File:** `src/AnalogLFO.cpp:1220, 1238, 1323, 1327-1330`
**Issue:** The 18HP refit duplicates the same conceptual font-scale family in two places (`drawTextOverlays` locals at 1327-1330 and `drawBpmStack` locals at 1220, 1238). They were bumped "in step" by hand this phase. There is no shared definition, so the "in step" relationship is enforced only by comment convention and is easy to break in a future retune. This is the root cause that produced WR-01.
**Fix:** Hoist the pill/font scale family to named `static constexpr` members of `WaveformDisplay` and reference them from both functions, so a single edit retunes consistently.

### IN-02: Layout anchors are unlabeled magic literals

**File:** `src/AnalogLFO.cpp:1539-1595`
**Issue:** Every widget anchor is a raw `mm2px(Vec(x, y))` literal. Column x-values (7.70 / 18.56 / 29.43 / 40.29 / 51.15) and row y-values (108.50, 119.50) repeat across many lines with no named constants, so a single mis-typed digit in a future edit would silently mis-place one control without any cross-check. This is idiomatic for VCV Rack panel code and was visually gated this phase, so it is informational only.
**Fix:** Optional — extract row baselines and the CV column pitch into named constants (e.g. `kCvRowY`, `kTrimRowY`, `kCol[5]`) so the trimpot row and jack row provably share columns by construction rather than by matching literals.

### IN-03: `ForgeKnobUtility` struct is now dead but still carries the bg-centering edit

**File:** `src/AnalogLFO.cpp:1482-1495`
**Issue:** Plan 03 upgraded RATE/PHASE from `ForgeKnobUtility` to `ForgeKnobSecondary`, and the summaries confirm zero `ForgeKnobUtility` instantiations remain. The struct (including its new `bg->box.pos` centering line added this phase) is dead code. Keeping it is harmless but adds an unused, separately-maintained code path that must be kept in sync with the other three knob structs.
**Fix:** Remove the `ForgeKnobUtility` struct if no future plan reintroduces a utility-knob tier; otherwise leave a one-line comment noting it is intentionally retained for a planned control.
